软件工程概述

软件危机的产生, 表现以及应对

定义

  • 软件危机时指在计算机软件的开发和维护过程中受到的一系列严重问题.

产生原因

  • 技术原因
    • 软件规模越来越大
    • 软件复杂度越来越高
  • 管理原因
    • 软件开发需要正确的理论指导, 过分依靠个人技巧与创作性
    • 对用户需求没有完整准确的认识

表现

  • 软件开发成本日益增长
  • 开发进度难以控制
  • 软件质量差
  • 开发过程无法有效介入和管理
  • 代码难以维护 等

应对方案

  • 软件工程

具体途径

  • 对计算机软件正确认识
  • 推广使用开发软件成功的技术和方法, 研究探索更好更有效的技术和方法, 消除错误的概念与做法
  • 开发和使用更好的软件工具
  • 对于时间, 人员, 资源等需要引入更加合理的管理措施

软件工程正是从技术和管理两方面研究如何更好的开发和维护计算机软件的一门新兴学科

软件工程知识体系

  • 开发过程
    • 软件需求, 设计, 构造, 测试, 维护
  • 支持过程
    • 配置管理, 工程管理, 软件工程, 工具与方法, 软件度量
  • 主要目标: 高效开发高质量软件, 降低开发成本

统一建模语言

  • 统一建模语言(UML)提供了一整套对系统建模的基础设施, 包括模型的表示及建模的方法等, 可以适用不同的系统层次. 它与其说是一种语言, 不如说是一种方法
  • UML将软件开发过程中语言表示与过程进行了分离

  • 包括的模型(UML2.0)

    • 用例图: 用于系统与使用者之间的交互
    • 类图: 类, 接口及其之间关系
    • 活动图: 系统中顺序和并行的活动
    • 对象图: 类图对应对象的实例
    • 还有: 顺序图, 通信图, 时序图, 交互概况图, 组成结构图, 组件图, 包图, 状态图, 部署图

  • 功能

    • 可视化 Visualization
    • 规格说明 Specification
    • 构造 Constructing
    • 文档化 Documenting
  • Kruchten的4+1视图
    • 逻辑视图
    • 进程视图
    • 开发视图
    • 物理视图
    • 用例视图
  • 系统开发的解空间: 基于用例的高度迭代的软件开发过程
    1. 用例建模: 描述系统功能性需求, 每个用例需要有一个用例规约
    2. 分析与设计: 要迭代式的进行系统的静态建模和动态建模
    3. 架构建模: 设计系统的软件体系结构
    4. 实现: 软件架构模型映射到可部署实现的模型解空间中

软件工程的开发方法

  • 方法: “如何做”
    • 传统方法
      • 生命周期模型
      • 划分为若干阶段
      • 每个阶段有各自任务
      • 阶段之间存在顺序
    • 面向对象方法
      • 对象作为融合数据及在数据之上的操作行为的统一的软件构件
      • 把所有对象都划分为类, 每个类存在一组数据与操作
      • 若干个类组成层次结构(亦称类等级), 下类继承自上类
      • 对象间仅能通过发送消息相互联系, 即封装性
  • 工具: “用什么做”
  • 过程: “如何控制, 协调, 保证质量”

面向对象方法的特点

  • 尽可能模拟人类习惯的思维方式
  • 是一个主动地多次反复迭代的演化过程
  • 概念与方法存在一致性, 阶段见平滑(无缝)过渡
  • 特殊到一般的归纳思维, 一般到特殊的演绎思维(继承的思想)
  • 降低了复杂性, 提高了可理解性
  • 对象, 以及面向对象的方法, 促进了软件的可重用性

软件开发过程

  • 软件开发过程(software development process), 亦称软件开发生命周期(SDLC, software development life cycle), 是软件开发的任务框架以及规范, 可以简称为软件生命周期与软件过程

软件生命周期与软件工程

  • 生命周期模型较一般化, 软件开发过程较具体化
  • 过程改进: 可重复的, 可预测的过程, 提高软件效率及质量
  • 生命周期是软件开发的宏观上的框架, 软件过程则涉及到软件开发的流程等管理细节, 在框架稳定的前提下允许对软件过程进行裁剪, 有以下4种:
    • 顺序式
    • 迭代式
    • 增量式
    • 敏捷式
  • 过程管理一般采用分治的思想, 软件生命周期可划分为:
    • 软件定义
    • 软件开发
    • 运行维护
