软件工程复习

发表于 2019-12-31 更新于 2020-01-08 分类于 软件工程 阅读次数： Valine：

软件工程概述

统一建模语言

  • 统一建模语言(UML)提供了一整套对系统建模的基础设施, 包括模型的表示及建模的方法等, 可以适用不同的系统层次. 它与其说是一种语言, 不如说是一种方法
  • UML将软件开发过程中语言表示与过程进行了分离
  • 包括的模型(描述系统的所有元素的集合以及元素之间的关系)
    • 用例图: 用于系统与使用者之间的交互
    • 类图: 类, 接口及其之间关系
    • 活动图: 系统中顺序和并行的活动
    • 对象图: 类图对应对象的实例
    • 还有: 顺序图, 通信图, 时序图, 交互概况图, 组成结构图, 组件图, 包图, 状态图, 部署图
  • 功能
    • 可视化 Visualization
    • 规格说明 Specification
    • 构造 Constructing
    • 文档化 Documenting
  • 视图: 每个视图捕获系统的一个方面, 图是通往模型的窗口
  • Kruchten的4+1视图
    • 逻辑视图
    • 进程视图
    • 开发视图
    • 物理视图
    • 用例视图
  • 系统开发的解空间: 基于用例的高度迭代的软件开发过程
    1. 用例建模: 描述系统功能性需求, 每个用例需要有一个用例规约
    2. 分析与设计: 要迭代式的进行系统的静态建模和动态建模
    3. 架构建模: 设计系统的软件体系结构
    4. 实现: 软件架构模型映射到可部署实现的模型解空间中

面向对象方法的特点

  • 尽可能模拟人类习惯的思维方式
  • 是一个主动地多次反复迭代的演化过程
  • 概念与方法存在一致性, 阶段见平滑(无缝)过渡
  • 特殊到一般的归纳思维, 一般到特殊的演绎思维(继承的思想)
  • 降低了复杂性, 提高了可理解性
  • 对象, 以及面向对象的方法, 促进了软件的可重用性

传统方法的特点

  • 生命周期模型
  • 软件划分为若干个阶段
  • 每个阶段有各自的任务
  • 阶段之间有某种顺序性

软件开发过程

  • 软件开发过程(software development process), 亦称软件开发生命周期(SDLC, software development life cycle), 是软件开发的任务框架以及规范, 可以简称为软件生命周期与软件过程

软件生命周期与软件工程

  • 生命周期模型较一般化, 软件开发过程较具体化
  • 过程改进: 可重复的, 可预测的过程, 提高软件效率及质量
  • 生命周期是软件开发的宏观上的框架, 软件过程则涉及到软件开发的流程等管理细节, 在框架稳定的前提下允许对软件过程进行裁剪, 有以下4种:
    • 顺序式
    • 迭代式
    • 增量式
    • 敏捷式
  • 过程管理一般采用分治的思想, 软件生命周期可划分为:
    • 软件定义
    • 软件开发
    • 运行维护

各阶段活动及任务

1.可行性分析与开发计划

描述可行性并拟定开发计划

  • 目的: 用最小的带价在尽可能短的时间内确定能否开发, 是否值得开发, 并提供依据(较高层次的需求分析与设计)
  • 任务: 确定项目规模, 目标, 约束以及限制, 进行简要需求分析以抽象出逻辑结构, 建立逻辑模型, 在经压缩设计 探索出若干的可供选择的主要解决方案
  • 可行性存在三个方面
    • 技术可行性
    • 经济可行性
    • 社会可行性

2.需求分析

对软件未来需要的功能进行详细分析, 同时随着项目进展, 需求分析要进行变更管理

  • 应充分理解和掌握用户对目标软件的期望, 除功能需求外还要对非功能性需求加以识别和分析
  • 输出是一份需求规格(Specification)说明书的文档

3. 软件设计

在需求分析的基础上寻求系统求解的框架

  • 分为
    • 概要设计
    • 详细设计

4. 程序编码

5. 软件测试

编码过程对代码严密测试, 发现设计过程的问题并及时纠正

  • 分为
    • 单元测试
    • 集成测试
    • 系统测试
  • 方法
    • 黑盒测试
    • 白盒测试
  • 需要编写测试方法, 测试用例, 分析结果, 保证有效性, 避免随意性

6. 软件维护

生命周期中持续时间最长的一段, 适应用户需求, 延长使用寿命

  • 改正性维护: 诊断并改正使用过程中发现的软件错误
  • 适应性维护: 修改软件来适应环境的变化
  • 完善性维护: 根据用户的要求改进或扩充软件使其更完善
  • 预防性维护: 修改软件来为将来的维护做准备

传统生命周期模型

瀑布模型

  • 特点:

    • 阶段间存在顺序性和依赖性, 文档驱动
    • 推迟实现, 不急于写代码
    • 质量保证的观点

  • 问题

    • 不希望有”变化”
    • 变化越晚, 代价越高

快速原型模型(Rapid Prototype)

  • 在用户和开发者之间建立”桥梁”的作用
  • 特点:
    • 要求对系统进行简单快速的分析, 快速构建软件模型
    • 用户与开发者试用或演示时加强沟通
    • 适合一个全新系统的开发
    • 可以尝试使用未来系统中需要的新技术
  • 问题
    • 所选开发技术与根据不一定是项目所需
    • 不断修改质量较差, 并且一般会抛弃之前建立的系统

喷泉模型

  • 典型的面向对象的生命周期模型
  • 体现了面向对象开发过程中的迭代和无缝特性
  • 把一个线性过程作为总目标

SCRUM

过程
  • 注重过程
  • 需求被定义为产品需求积压(Product Backlogs)
  • 开发过程分为多个冲刺(Sprint)
  • 燃尽图为公开展示的, 冲刺过程中显示当前未完成的数目
  • 每个冲刺之前开始之前, 要举行Sprint计划会议, 每天有Scrum会议, Sprint结束后又Sprint评审会议, 最后有一个回顾会议
角色
  • 产品拥有着(Product Owner): 远景规划, 平衡相关利益, 确定开发团队与客户, 开发团队与客户的联络点
  • 利益收集者(Stakeholder): 客户代表
  • 专家(Scrum Master): 开发团队与产品拥有者的联络点
  • 团队成员: 开发人员

DevOps

  • 核心目标: 自动化, 可持续交付
  • 纵向集成打通了应用全生命周期
  • 横向集成打通了架构, 开发, 管理, 运维等部门墙

类的分析与设计

==分析(领域)类图==

管理类与控制类

  • 管理类 为对同类对象的协调与管理, 主要负责同类对象的创建, 代理访问其他对象的信息等. 管理类对于其所管辖的所有对象都要有统一的处理方式, 一般借助容器实现
  • 控制类 对一个或几个用例所特有的控制进行建模, 控制类通常控制和协调不同对象的行为, 用来封装用例的特殊行为

设计优化

设计原则

  • 接口隔离: 尽量继承接口, 并在实现时关注需要的部分, 即把不需要的隐藏
  • 依赖倒置: 尽量实现抽象, 依赖一个抽象类要比依赖一个实体类好得多
  • 开放封闭: 模块对于修改是封闭的, 对于扩展是开放的
  • Liskov替换原则: 任何父类出现的地方替换为子类不会出错
  • 单一职责: 类的功能应该只有一个
  • 合成/聚合复用: 尽量使用合成/聚合形式的委托重用, 尽量不使用继承重用(禁止多重继承)

设计模式

创建模式

  • 抽象工厂模式
    • 解耦客户类在创建产品类时引入的耦合
    • 工厂提供一系列相关, 相互依赖的接口, 客户无需指定具体产品即可创建
  • 单例模式
    • 保证一个类仅有一个实例, 提供一个它的全局访问点
    • 构造方法私有
    • 共有的取得实例
    • 实例变量为私有的或者收到保护的

结构模式

  • 适配器模式: 将原来两个不匹配的接口通过适配器来能够相互工作
  • 门面模式: 要求外部与一个子系统的通信必须通过一个统一的门面对象进行
  • 代理模式: 一般用于价值(稀缺)资源的管理(如数据库连接), 由代理对象控制连接来提高效率

行为模式

  • 观察者模式: 一对多的依赖, 多个观察者监听一个主题对象, 发生变化时及时通知
  • 策略模式: 针对复杂问题的不同情况有不同策略, 不会影响客户端
  • 状态模式: 策略模式的一种应用, 状态改变时产生行为, 行为包装在不同的状态对象里, 每个状态对应抽象状态的子类

软件测试

软件度量

McCabe

  • 控制流图
  • McCabe, 计算方法为: V(G) = 边数-节点数+2
  • McCabe值应当尽量降低, 可以
    • 优化算法结构
    • 分解if嵌套
    • 利用多态

LCOM

LCOM* = [(1/实例变量数)*(访问每个实例变量的方法数值和) - 方法数]/1-方法数

  • LCOM*为0时, 内聚性最好

基于控制流的测试(覆盖指标)

  • 语句覆盖 = 控制流图中经过的节点数/所有节点数
  • 分支覆盖 = 控制流图中经过的边数/所有边数
  • 条件覆盖 = (取值为真的原子谓词 + 取值为真的原子谓词) / 2 * 所有原子谓词数
  • 多条件组合覆盖 同上, 但要考虑组合条件
  • 路径覆盖: 计算McCabe环形复杂度, 导出测试用例
